Policy Builder — what I'd change, and a proposed redesign

The current Builder asks supervisors to author an attendance rulebook through four equal-weight columns sitting inside a nested-card stack, with no breadcrumb back to the policy list, no save state, and help text that's longer than the inputs themselves. The redesign below treats the policy as a document — name and status at the top, one section at a time, a sticky table-of-contents that doubles as a progress map, and compound inputs that group rules instead of splitting them across columns.

Sidebar & navigation

  • Expose Policy Studio's sub-tree. Today the only way back to the list of policies is the browser back button. Show Policies / Templates / Builder / Activity log nested under Attendance > Policy Studio, with the active page highlighted.
  • Search + ⌘K. The nav is four levels deep; a search field at the top of the sidebar lets you jump straight to "Policy STD_8_5" without expanding three parents.
  • Demote "Pinned". "Database Tables" and "Claim Assignments" aren't pinned for an attendance admin — they're org-level shortcuts. Move them below the active workspace or behind a toggle.

Page header

  • Title is the policy, not the tool. The current page reads "Policy Builder" with the policy name buried inside a card title. Promote Standard 8 to 5 to the H1 and demote "Policy Builder" to the breadcrumb.
  • Status pill next to the title. Draft / Active / Archived needs to be visible from the page header so it's obvious whether you're editing a live policy or a draft.
  • Meta strip. Code, effective date, last edit, and "used by N rosters" sit under the title — answers the question "is it safe to change this?".
  • Actions at the top. Preview, Validate, Publish — plus an "Autosaved 3 min ago" indicator. The current page has no save affordance at all.

Form layout

  • Drop the parent card. Three nested cards inside one parent card make the borders compete. Use flat sections separated by spacing — one card per group.
  • Group identity, separate timing. Today's four-up grid mixes Policy Code & Name (identity) with Effective From & Status (lifecycle). Identity goes at the top; lifecycle moves to its own section near the bottom where supervisors expect activation controls.
  • Sticky TOC with completion dots. Long forms need a map. The left rail shows section status (done / warning / empty) and doubles as anchor links — a supervisor returning to a half-finished policy lands on the first unfinished section.

Inputs & help text

  • Combine rounding mode + minutes. "Daily Rounding" and "Minutes" are one rule split across two columns. Merge into a compound input: [ Nearest ▾ ] [ 15 ] min. Two columns of help text collapse into one.
  • Number fields get unit suffixes. Every numeric field should carry its unit inside the field (min, hours, × base) so a value of "60" can't be misread as seconds or hours.
  • Status becomes a segmented control. Three mutually-exclusive values — Draft / Active / Archived — don't deserve a dropdown. A segmented switch shows the full state at a glance.
  • Tighten help text. "Plainly: the nickname supervisors will recognize" reads as filler. Aim for one short sentence per field; move longer explanation into an i tooltip. Drop "BLB" — supervisors don't need the internal product code in microcopy.
  • Show a worked example. For rounding fields, append a one-liner: "7h 52m worked → 8h 00m on the timesheet." Concrete beats abstract every time.

Missing affordances

  • "Test against shift" tool. The subtitle promises validation before supervisors use the policy. Add a small panel that runs the current rules against a sample clock-in to surface the resulting timesheet line — the fastest way to catch an inverted rounding rule.
  • "Duplicate from policy". Most new policies fork an existing one. A duplicate action on the Policies index avoids re-entering 20 fields.
  • Activity log + diff. When a published policy is edited, supervisors want to know what changed and when. The fourth Policy Studio sub-item.
